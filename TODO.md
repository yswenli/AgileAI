## A. Prevent repeated skill prompt accumulation in continuous conversations

### Goal
Ensure local skill system prompts are not duplicated across multi-turn conversations.

### Problems to solve
- `PromptSkillExecutor` currently injects a skill system prompt for each execution
- If callers reuse returned history directly, the same skill prompt can accumulate repeatedly
- Repeated prompt accumulation increases token usage and can distort behavior over time

### Tasks
- [x] Define a canonical way to mark a skill system message as generated by AgileAI
- [x] Detect and avoid reinserting the same skill prompt into history
- [x] Decide whether skill prompt should live in transient execution context vs persisted conversation history
- [x] Add tests for multi-turn execution without duplicated skill system prompts
- [x] Verify no regression for one-shot skill execution

### Suggested implementation notes
- Add metadata or a deterministic marker for skill system messages
- Filter or deduplicate before dispatching the next request
- Keep user/assistant/tool messages intact while removing duplicated framework-generated prompts

## C. Add active skill continuation policy

### Goal
If a session already has an active skill, continue using it by default unless there is a strong reason to switch.

### Why this matters
Without continuation, every new turn must be replanned from scratch, which is unnatural for multi-turn skill workflows.

### Tasks
- [x] Define `ISkillContinuationPolicy` or equivalent policy abstraction
- [x] Continue active skill when session state already has one
- [x] Fall back to planner only when:
  - [x] there is no active skill
  - [x] user explicitly changes topic/skill
  - [x] a policy decides continuation confidence is too low
- [x] Decide how a skill is exited/cleared
- [x] Add tests for:
  - [x] same-skill continuation across turns
  - [x] switching from one skill to another
  - [x] clearing active skill and returning to plain chat

### Suggested implementation notes
- First version can use a simple policy:
  - continue active skill by default
  - switch only on explicit preference or strong planner match
- Keep the policy deterministic before introducing LLM-based switching heuristics
